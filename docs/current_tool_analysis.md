# 現行ツール分析（課題整理と改善提案）

## 1. 分析の前提
- 本分析は、要件定義書・実行計画・現行実装（UI/API/可視化ロジック）の差分を確認して整理した。
- 特に、依頼のあった「情報共有ページ（グループ活動）」と「可視化ツール（UMAP）の効率化」を優先して検討した。

---

## 2. 要件とのギャップ（優先度順）

### P0（早急対応）

#### 2-1. 情報共有ページ（グループ活動）が未実装
- **要件側**では「斬新・面白いと感じたことを選択・共有するグループ活動」が明記されている。
- **データモデル**には `GroupActivity` がある一方、現行コードには `group-activities` API と該当UIが見当たらない。
- **影響**：Big Five → テーマ → 質問回答の間にある「共有フェーズ」が欠け、学習体験（個人→集団への接続）が弱い。

**提案**
1. `POST /api/group-activities` / `GET /api/group-activities` を追加。
2. 生徒向けに「共有ページ」を追加（複数選択 + 任意自由記述 + 送信確認）。
3. 管理者向けダッシュボードに、共有結果の集約（頻出トピック、引用、時系列）を表示。

#### 2-2. エクスポートAPIに認証・認可チェックがない
- `sessionId` を知っていれば、JSON/CSVを取得できる実装になっている。
- **影響**：教育データ（Big Five・回答）の漏えいリスクが高い。

**提案**
1. `getAuthContext` と `requireAuth(..., 'admin')` を必須化。
2. `sessionId === auth.sessionId` の検証を追加。
3. エクスポート操作を監査ログ対象にする（将来の `AuditLog` 活用）。

---

### P1（機能品質向上）

#### 2-3. UMAP計算が毎回クライアント再計算で、再利用・再現性が低い
- 可視化ページ表示のたびにブラウザ側で UMAP + K-means を実行している。
- スキーマには `OpinionMapping` があるが、現状は保存・再利用されていない。
- **影響**：参加者が増えると表示待ちが伸びる、教員と生徒で見えるクラスタが微妙に揺れる可能性がある。

**提案（効率化）**
1. サーバー側でバッチ計算し、`OpinionMapping` に保存（セッション単位キャッシュ）。
2. 生徒画面は保存済み座標を取得する方式に変更。
3. 回答更新時のみ再計算（差分トリガー）して不要計算を削減。

#### 2-4. K-means初期値が先頭k件固定で、クラスタ品質が不安定
- コメントは「ランダム初期化」だが実装は `points.slice(0, k)`。
- **影響**：データ順序依存が強く、クラスタ結果が歪む可能性。

**提案**
1. K-means++ 初期化を導入。
2. 複数回試行して SSE（誤差平方和）が最小の結果を採用。
3. 可能なら `k` 自動推定（シルエット係数）を導入。

#### 2-5. 「25次元」前提と実データ次元がズレる可能性
- 要件は「20問 + 5特性 = 25次元」だが、実装は「そのテーマの質問数 + 5特性」。
- **影響**：テーマごとに次元が変わり、比較の一貫性が崩れる。

**提案**
1. MVPとしては「常に20問に正規化」する（不足分は0埋め）。
2. あるいは要件を更新し「テーマ依存次元」を正式仕様にする。

---

### P2（体験改善）

#### 2-6. 可視化の読解支援が最小限
- 要件で想定される「クラスタ説明」「対立・合意領域の検出」「チュートリアル」は未実装または限定的。
- 現状は散布図 + 簡易内訳 + 全体分布が中心。

**提案**
1. クラスタ説明カード（代表的設問・回答傾向）を追加。
2. 対立領域（クラスタ間距離）・合意領域（高密度領域）をオーバーレイ表示。
3. 初回のみ可視化チュートリアルを表示（「距離=類似」「優劣ではない」）。

#### 2-7. 情報共有→可視化の接続不足
- 共有ページが無いため、「生徒が投稿した観点」が可視化に反映されない。
- **提案**：共有トピックをタグ化し、クラスタごとの頻出タグを可視化パネルに連携。

## 3. UMAP効率化の実装ロードマップ（現実的な順序）

1. **Step 1: キャッシュ導入（最小実装）**  
   セッションごとに最新 `OpinionMapping` を保存し、可視化画面はまずキャッシュを読む。

2. **Step 2: 再計算トリガー制御**  
   新規回答が入ったときだけ再計算フラグを立てる（閲覧時の常時計算を廃止）。

3. **Step 3: 品質改善**  
   K-means++、複数シード試行、メタデータ（使用パラメータ・生成時刻）を保存。

4. **Step 4: 読解支援拡充**  
   対立/合意領域、クラスタ要約、注釈UIを段階的に追加。

---

## 4. 情報共有ページ（グループ活動）の最小要件案

### 生徒向け
- 「共感した観点」を複数選択（例: 倫理、格差、自由、安全性、環境）。
- 任意テキスト（140〜300字目安）。
- 送信後は編集可能（授業時間内のみ）。

### 管理者向け
- 投稿数、未投稿者、頻出タグ、代表コメント表示。
- NGワード・個人特定情報の簡易フィルタ。

### データ連携
- `GroupActivity.selections` をタグ配列として保存。
- 可視化パネルで「クラスタ別頻出タグ」を表示し、議論導線に使う。

## 5. 直近2スプリントの推奨タスク

### Sprint A（安全性 + 基盤）
1. export API の認可保護。
2. group-activities API の追加。
3. 生徒の情報共有ページ（最小UI）。

### Sprint B（可視化品質）
1. `OpinionMapping` 保存/再利用。
2. K-means初期化改善（K-means++）。
3. 可視化サイドパネルにクラスタ要約追加。

---

## 6. 期待効果
- **学習体験の一貫性**：個人回答→共有→可視化→対話の流れがつながる。
- **運用性**：再計算削減で授業中の待ち時間を短縮。
- **説明可能性**：クラスタ説明と注記強化で誤解（優劣判断）を抑制。
- **安全性**：エクスポート保護で機微データ管理を改善。
