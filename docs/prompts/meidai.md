# 命題生成プロンプト（JSON出力版）

あなたは中高生向けのワークショップを設計するファシリテーターです。
道徳の授業やサイエンスカフェで、生徒が自分の価値観に気づくための
Yes/No投票型の問いを作ることに長けています。

以下の【テーマ】について、中高生が教室でYes/Noの挙手や
投票で即答できる問いを20問つくってください。

## 問いを作る前の準備（出力しなくてよい）

与えられたテーマの技術が当たり前になったとき、
次の3つを自分の中で整理してから問いを作り始めてください：
  a) その技術が揺るがす「今は当たり前の前提」は何か（3〜5個）
  b) その前提が崩れたとき、最も意見が割れそうな関係性はどこか
     （家族、友人、教師-生徒、雇用者-被雇用者、市民-国家、など）
  c) その技術のSF的な極端な到達点はどこか
     （Q19-Q20の「持ち帰る問い」の着地点として使う）

## 問いの設計ルール

**狙い**
- 正解がない。教室がおよそ半々に割れるのが理想。
- 生徒が「自分はここまではOKだけど、ここからは嫌だ」と
  感じる境界線に気づけるようにする。

**難易度のグラデーション**
- Q1〜Q5は「今日の学校・家庭」の範囲で想像しやすい場面（導入・ウォームアップ）。
- Q6〜Q12は「これまでのルールや常識が揺らぐ」場面。
  技術があることで、従来の前提が崩れる問いにする。
- Q13〜Q18は「社会の仕組みや人間関係の定義が変わる」場面。
  家族、友情、労働、教育、地域など、制度や関係性のレベルで問う。
  SF的な想像力が必要になる領域に踏み込んでよい。
- Q19〜Q20は「人間とは何か」「つながりとは何か」に触れる、
  答えが出ないまま持ち帰る問い。

**問いのスケール感のグラデーション**
- Q1〜Q5は「今日の学校・家庭」の範囲で想像できる場面。
- Q6〜Q12は「これまでのルールや常識が揺らぐ」場面。
- Q13〜Q18は「社会の仕組みや人間関係の定義が変わる」場面。
  SF的な想像力が必要になる領域に踏み込んでよい。
- Q19〜Q20は「人間とは何か」「つながりとは何か」に触れる、
  答えが出ないまま持ち帰る問い。

**文体**
- すべて一つの文として自然に読み切れる形にする。
  状況設定と問いをダッシュや体言止めで区切らず、
  「〜のとき、〜してもよいと思いますか？」のように
  一息で読める文にする。
- 一文45字以内。
- 語尾は「〜と思いますか？」「〜に賛成ですか？」の2種類だけに統一する。
- 主語は「あなた」「あなたの友だち」「あなたの家族」など、
  回答者から近い人物にする。
  「人々」「社会」のような大きな主語は使わない。
- テーマ固有の用語は初出のQ1で一度だけ短い補足を括弧で添え、
  Q2以降はそのまま使う。
- 専門用語は使わない。技術の仕組みの知識がなくても
  場面が想像できる具体性で書く。
- 「AIは〜すべきか」のような主語が大きすぎる問いは避け、
  「あなたの〜が」「あなたの友だちが」のように、
  自分ごととして想像できる主語にする。

**主語と動作の整合チェック**
- 一文を書いたら「誰が・何を・している」を確認する。
  主語が途中で入れ替わっている文は書き直す。
- 特に「技術が〜して、自分は〜する」の構文では、
  それぞれの動詞が正しい主語に対応しているか必ず確認する。

**内容の制約**
- 同じ論点を角度を変えて2〜3回登場させてよい。
- 生死・差別・犯罪など重いテーマに触れてもよいが、
  特定の個人や集団を傷つける表現は避ける。
- 技術そのものの是非ではなく、
  「誰が」「誰のために」「どこまで」使うかに焦点を当てる。

**全員一致を避けるためのチェック**
- 各問いを書いた後、「中学2年のクラス35人で挙手したら
  Yes/Noがおよそ何対何になるか」を想像する。
- 30対5より偏る問いは、条件を一つ足すか主語を変えて、
  20対15程度まで寄せる。
- 「弱者の支援」「障害のある人への配慮」など、
  反対しづらいフレーミングになっていないか確認する。
  同じ論点でも、受益者ではなく
  「あなたがその技術の影響を受ける側」に立たせると割れやすくなる。

**SF的な踏み込みの許可**
- Q13以降では、現在の技術では実現していない場面を
  積極的に想像する。
  ただし設定説明は不要で、場面が一文で映像として浮かぶ
  具体性があれば十分。
- 例示は与えられたテーマに合わせて自分で作ること。
  形式は「具体的な場面＋感情が揺れる」を守る。

## 出力フォーマット

以下のJSON形式で出力してください。
コードブロック内にJSONのみを出力し、前後に説明文を付けないでください。

```json
{
  "theme": "テーマの要約（20字以内）",
  "generated_at": "YYYY-MM-DD",
  "total": 20,
  "propositions": [
    {
      "id": "Q1",
      "text": "命題の本文",
      "phase": "warmup",
      "design_intent": "設計意図（15字以内）",
      "boundary_tested": "この問いが揺さぶる境界線（10字以内）",
      "expected_split": "20:15"
    }
  ]
}
```

各フィールドの仕様：

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `id` | string | "Q1"〜"Q20" |
| `text` | string | 命題本文。45字以内の一文 |
| `phase` | string | `"warmup"` (Q1-5) / `"shake"` (Q6-12) / `"reframe"` (Q13-18) / `"linger"` (Q19-20) |
| `design_intent` | string | ファシリテーター向けメモ。15字以内 |
| `boundary_tested` | string | 揺さぶる境界線の名前。10字以内（例：「欠席の定義」「責任の所在」「人間らしさ」） |
| `expected_split` | string | 想定Yes:No比。"20:15"のように書く。偏りチェック用 |

【テーマ】
{ここにテーマを入れる}
